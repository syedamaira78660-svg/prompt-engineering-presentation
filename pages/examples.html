Chapter 07 · Taste Test

What You Ordered vs. What You Meant

Three orders, the naive way they usually get placed, and the engineered version that actually gets you what you wanted. Click "Show the improved order" on each to reveal it — notice which Golden Rule each addition demonstrates.

Case Study 1 · Technical & Coding Tasks

Scraping a website

The naive version usually needs 3–4 follow-up prompts to fix bugs, add error handling, and stop assuming a perfect environment.

what you ordered
Write a python script to scrape a website.

Resulting AI behavior: basic, un-commented code, no error handling, assumes a perfect environment.

what you meant
Act as a senior Python developer experienced in web scraping and data pipelines.

Write a Python script that scrapes article titles and publish dates from a news website's homepage.

I'll run this locally with Python 3.11 and can install packages via pip. The site uses static HTML — no JavaScript rendering needed.

Use requests and BeautifulSoup only. Respect robots.txt. Handle network errors and missing elements gracefully. Add a 1-second delay between requests if scraping multiple pages.

Deliver a single .py file with inline comments, plus a short usage example at the bottom.
Role · senior Python developer Task · scrape titles + dates Context · Python 3.11, static HTML Constraints · robots.txt, error handling Format · single commented .py file

Case Study 2 · Content & Writing Tasks

A blog post about productivity

The naive version reads like every other "5 tips to be more productive" listicle on the internet.

what you ordered
Write a blog post about productivity.

Resulting AI behavior: clichés, generic bullet points, no clear audience or voice.

what you meant
Act as a productivity coach who writes for early-career software engineers.

Write a 600-word blog post titled "Why Your To-Do List Is Lying to You," arguing that time-blocking beats task lists for deep work.

My audience is junior developers who feel busy but unproductive, juggling Slack, meetings, and coding.

Use a conversational tone, no corporate jargon. Include one concrete example from a typical dev's day. Avoid generic "wake up at 5am" advice.

Format as a Markdown blog post with a hook opening line, three subheadings, and a one-line takeaway at the end.
Role · productivity coach Task · 600-word post, specific angle Context · audience = junior devs Constraints · tone, no clichés Format · Markdown, 3 subheadings

Case Study 3 · Data & Business Analysis

Diagnosing a sales drop

The naive version produces vague commentary that never actually references your numbers.

what you ordered
Analyze this sales data and tell me what's wrong.

Resulting AI behavior: "sales seem to be declining" — generic, no structure, no actionable recommendation.

what you meant
Act as a senior business analyst who specializes in retail sales trends.

Analyze the attached quarterly sales data and identify the top 3 factors likely driving the 12% revenue drop in Q3.

We're a mid-size e-commerce company selling home goods. Q3 included a pricing change and a shipping delay in August.

Base conclusions only on the provided data, and flag any assumptions explicitly. Keep it under 400 words.

Present the findings as a Markdown table (Factor, Evidence, Confidence) followed by 3 recommended next steps.
Role · senior business analyst Task · find top 3 drivers Context · pricing change, shipping delay Constraints · data-only, 400 words Format · table + next steps
